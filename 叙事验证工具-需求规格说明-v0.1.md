# 叙事验证工具需求规格说明

版本：v0.1  
状态：待用户评审  
适用范围：`验证工具` 文件夹内的验证性小程序、原型、样板数据和后续脚本  
工程策略依据：先稳定需求与对象模型，再做交互原型和实现；正式小说文件只读引用，不直接污染。

## 1. 需求背景

当前小说项目已经存在多种支持文档，包括正文、分章大纲、分章简纲、事件脉络、势力目标线、人物梗概、时间线统计、碎片池和若干视角备选材料。这些文档各自有价值，但问题在于：

1. 整体故事推进方向不够直观。
2. 多条事件线、目标线、信息线之间的关系需要人工反复查找。
3. 单个章节内部逻辑是否闭合，不容易快速验证。
4. 章节写完后，哪些结构事实应同步、哪些仍待确认，缺少明确检查流程。
5. 新线索是否该加入、并入、延后或废弃，缺少统一判断工具。

本工具的目标不是替代写作，也不是自动生成正文，而是提供一个“故事事件流与章节函数验证工具”。

## 2. 核心定位

### 2.1 主入口

工具主入口必须是“全局故事事件流”，不是某一个章节。

主视角应回答：

- 整个故事发展到哪个阶段？
- 主线和支线分别推进到哪里？
- 当前阶段的叙事压力是什么？
- 哪些目标线正在变化？
- 哪些信息已经暴露，哪些仍被隐藏或误解？
- 下一阶段需要承接什么？

### 2.2 章节定位

章节不是主轴。章节是全局事件流中的一个可展开节点。

每个章节被视为一个“函数”：

```text
章节输出 = 章节函数(章节输入)
```

其中：

- 章节输入：上游事件、人物状态、已知信息、未解决压力、必须承接的线索。
- 章节函数内部：场景、冲突、事件推进、目标变化、信息流动、情绪变化和风险制造。
- 章节输出：下游状态、被推进的线、被改写的目标、暴露或隐藏的信息、新风险和同步建议。

工具需要帮助使用者拆开章节这个黑盒，验证其内部逻辑是否成立。

## 3. 产品目标

### 3.1 总体目标

建立一个本地可打开、可点击、可验证的叙事工作台，让作者先看全局故事流，再下钻到事件群、线索、角色目标和章节函数。

### 3.2 直接目标

1. 用全局故事地图呈现整部小说的阶段主轴。
2. 用事件流视图呈现各阶段中的事件群和因果链。
3. 用线索流视图呈现主线、支线、目标线和信息线的推进状态。
4. 用章节函数视图拆解具体章节的输入、内部逻辑和输出。
5. 用风险报告指出时间、因果、目标、信息流、章节功能和同步状态的问题。
6. 用工作流引导用户从“全局判断”进入“局部验证”，避免一上来铺满所有细节。

## 4. 非目标

当前阶段不做：

1. 不自动续写正文。
2. 不自动改写正式大纲、正文、事件脉络或势力目标线。
3. 不把规划中内容直接标为已确认。
4. 不以单章作为主入口。
5. 不做复杂数据库和多人协作系统。
6. 不从全书正文自动抽取全部事件。
7. 不把所有支持文档合并成一个大摘要。
8. 不用漂亮图表掩盖因果、目标和信息流问题。

## 5. 用户与使用场景

### 5.1 主要用户

主要用户是作者本人，以及协助作者进行结构分析、改写准备和连续性检查的写作助手。

### 5.2 典型场景

#### 场景 A：查看全局故事进度

用户打开工具后，先看到整部小说的阶段主轴和事件流。

用户需要知道：

- 当前故事处于哪个大阶段。
- 前一阶段留下了什么压力。
- 当前阶段正在推进哪些线。
- 下一阶段将接收哪些输出。

#### 场景 B：分析某一阶段

用户点击一个阶段，例如“会后缓冲期”或“流血冲突期”。

工具展示：

- 阶段事件群。
- 阶段入口状态。
- 阶段出口状态。
- 该阶段涉及的事件线、目标线、信息线。
- 阶段风险。

#### 场景 C：分析某一事件群

用户点击一个事件群，例如“会议结果被城市机器吸收”。

工具展示：

- 事件群内的关键事件。
- 事件之间的因果关系。
- 哪些角色目标被改写。
- 哪些信息从隐藏转为可见。
- 哪些章节承载这个事件群。

#### 场景 D：分析某一章节

用户从事件流或阶段视图进入章节函数视图。

工具展示：

- 章节输入。
- 章节内部运作。
- 章节输出。
- 本章风险。
- 本章与全局事件流的关系。

#### 场景 E：判断是否加入新线索

用户输入一个新线索想法。

工具辅助判断：

- 它解决哪个全局结构问题。
- 它属于已有线索还是需要独立成线。
- 它应该进入哪个阶段。
- 它需要哪几个章节承载。
- 它是否会抢走当前主线。
- 它加入后会改变哪些目标和信息流。

## 6. 信息架构

工具的信息层级必须是：

```text
全局故事地图
  -> 阶段视图
    -> 事件群视图
      -> 线索流 / 目标流 / 信息流
        -> 章节函数视图
          -> 风险报告 / 同步建议
```

禁止把章节放在第一层作为主入口。

## 7. 核心对象模型

### 7.1 StoryMap 全局故事地图

定义整部小说的全局结构。

字段：

| 字段 | 含义 |
| --- | --- |
| story_id | 故事编号 |
| title | 故事名称 |
| current_phase | 当前阶段 |
| dominant_line | 当前主导线 |
| phase_list | 阶段列表 |
| global_risks | 全局风险 |
| source_refs | 来源文件 |

### 7.2 Phase 阶段

阶段是全局故事流中的大段推进单位。

字段：

| 字段 | 含义 |
| --- | --- |
| phase_id | 阶段编号，例如 S0、S1、S2 |
| title | 阶段名称 |
| entry_state | 阶段入口状态 |
| exit_state | 阶段出口状态 |
| pressure | 阶段叙事压力 |
| event_groups | 事件群 |
| active_lines | 活跃线索 |
| risk_items | 风险项 |
| chapter_refs | 承载章节 |

### 7.3 EventGroup 事件群

事件群是多个事件组成的因果单元。

字段：

| 字段 | 含义 |
| --- | --- |
| group_id | 事件群编号 |
| title | 事件群名称 |
| phase_id | 所属阶段 |
| function | 结构功能 |
| entry_event | 入口事件 |
| exit_event | 出口事件 |
| events | 包含事件 |
| affected_lines | 被影响线索 |
| chapter_refs | 承载章节 |

### 7.4 Event 事件

事件是工具中最小的叙事推进节点。

字段：

| 字段 | 含义 |
| --- | --- |
| event_id | 事件编号，例如 E001 |
| title | 事件名称 |
| phase_id | 所属阶段 |
| group_id | 所属事件群 |
| time_ref | 时间位置 |
| chapter_refs | 关联章节 |
| participants | 参与人物 |
| event_lines | 事件线 |
| goal_changes | 目标变化 |
| info_changes | 信息变化 |
| output_state | 事件输出 |
| risk_items | 风险项 |
| status | 规划中、已成文、待确认、已确认 |

### 7.5 Line 线索流

线索流包括主线、支线、目标线、信息线。

在多轨时间轴界面中，必须进一步区分“事件线”和“人物条”：

- 事件线只记录文中实际发生、实际叙述到的事件段。
- 事件线上的每个事件块都代表一段被章节承载的叙事事实。
- 事件线不能用虚线或淡色范围表示“后台仍在执行但文中没有写出”的状态。
- 如果某个后台行动没有进入正文叙述，它不应扩大事件线事件块的时间范围。
- 事件线可以包含多个离散事件块；这些事件块共同构成该线的推进，而不是每个事件块都等于整条线的全部生命期。

字段：

| 字段 | 含义 |
| --- | --- |
| line_id | 线索编号 |
| line_type | 主线、事件线、目标线、信息线 |
| title | 线索名称 |
| current_state | 当前状态 |
| start_event | 起点事件 |
| latest_event | 最新事件 |
| next_expected_event | 下一预期事件 |
| linked_actors | 关联人物 |
| risks | 断裂、抢戏、重复、提前解决等风险 |

### 7.6 CharacterLane 人物条

人物条是某个人物在某条事件线下的参与视图，不是全局人物传记，也不是事件线本身。

人物条必须支持“实段”和“虚段”的区分：

- 实段：正文中明确写到的人物行为、出场、影响或被他人直接感知的结果。
- 虚段：人物理论上正在执行、维持状态、暗中影响或承担后台任务，但正文没有直接描写。
- 虚段不能替代事件线事件块；它只解释人物后台状态，不代表文中已经发生了新的叙事事件。
- 如果一个人物行为跨越多个章节，必须判断哪些部分被正文描写，哪些只是后台持续。
- 长人物条应优先拆成“实段 + 虚段”或多个离散实段，而不是把一个单一实框拖长。

字段：

| 字段 | 含义 |
| --- | --- |
| lane_id | 人物条编号 |
| actor_id | 人物编号 |
| parent_line_id | 所属事件线 |
| segment_list | 实段和虚段列表 |
| explicit_segments | 文中明确描写的行为段 |
| background_segments | 未直接描写但需要追踪的后台状态段 |
| evidence_refs | 支撑该人物段存在的章节、事件或设定来源 |
| risk_items | 误把后台状态当正文事件、人物行为过长、缺少触发等风险 |

### 7.7 CharacterSegment 人物条片段

人物条片段是人物条中的最小显示单元。

字段：

| 字段 | 含义 |
| --- | --- |
| segment_id | 片段编号 |
| actor_id | 人物编号 |
| parent_line_id | 所属事件线 |
| start_ref | 起始章节或事件 |
| end_ref | 结束章节或事件 |
| visibility | explicit 或 background |
| title | 片段名称 |
| function | 该片段在当前事件线里的作用 |
| trigger_event | 触发该片段的事件 |
| source_refs | 文中证据或规划来源 |
| risk_items | 片段风险 |

### 7.8 ActorGoal 角色目标

角色或势力的目标状态。

字段：

| 字段 | 含义 |
| --- | --- |
| actor_id | 人物或势力编号 |
| actor_name | 名称 |
| overall_goal | 总体目标 |
| current_attention | 当前注意力点 |
| current_goal | 当前目标 |
| trigger_event | 最近触发事件 |
| action_output | 行动输出 |
| side_effect | 副作用 |
| next_pressure | 下一步压力 |

### 7.9 InfoNode 信息节点

跟踪关键信息如何流动。

字段：

| 字段 | 含义 |
| --- | --- |
| info_id | 信息编号 |
| content | 信息内容 |

| known_by | 已知人物 |

### 7.10 ChapterFunction 章节函数

章节函数是局部分析对象，不是主入口对象。

字段：

| 字段 | 含义 |

## 8. 核心界面需求

### 8.1 全局故事地图页面

这是工具首页。

必须展示：

1. 全局阶段主轴。
2. 当前主导线。
3. 各阶段入口状态和出口状态。
4. 每个阶段涉及的主要线索。
5. 当前待验证风险。

用户操作：

- 点击阶段进入阶段视图。
- 点击线索进入线索流视图。
- 点击风险进入风险报告。
- 点击“继续验证”进入推荐工作流。

不得展示过多章节细节。章节只作为阶段承载信息出现。

### 8.2 阶段视图页面

用于查看某一阶段的结构。

必须展示：

1. 阶段入口状态。
2. 阶段事件群。
3. 阶段出口状态。
4. 活跃线索。
5. 目标变化摘要。
6. 信息流变化摘要。
7. 阶段风险。
8. 承载章节列表。

用户操作：

- 点击事件群进入事件群视图。
- 点击章节进入章节函数视图。
- 点击线索查看该线在本阶段如何推进。

### 8.3 事件群视图页面

用于查看一组事件如何组成因果单元。

必须展示：

1. 事件群结构功能。
2. 入口事件和出口事件。
3. 事件链。
4. 被改变的角色目标。
5. 被暴露或遮蔽的信息。
6. 关联章节。
7. 风险项。

用户操作：

- 点击事件查看事件详情。
- 点击目标变化查看角色目标卡。
- 点击信息变化查看信息节点。

### 8.4 线索流视图页面

用于查看某条线索在全局中的推进。

必须展示：

1. 线索起点。
2. 当前状态。
3. 已发生节点。
4. 未发生但已规划节点。
5. 断裂点。
6. 与其他线索的交叉点。
7. 是否有抢主线、重复或提前解决风险。

### 8.5 章节函数页面

用于拆解某一章这个黑盒。

页面必须分成五个区域：

#### 8.5.1 输入区

展示本章必须接收的上游状态：

- 上游事件。
- 上游人物目标状态。
- 上游信息状态。
- 上游风险。
- 必须承接的全局阶段位置。

#### 8.5.2 内部运作区

展示章节内部如何运行：

- 场景顺序。
- 本章事件。
- 冲突结构。
- 角色行动。
- 目标变化。
- 信息流动。
- 情绪变化。

#### 8.5.3 输出区

展示本章结束后必须产生的下游状态：

- 推进了哪些事件线。
- 改变了哪些目标线。
- 暴露了哪些信息。
- 保留了哪些悬念。
- 制造了哪些后续压力。

#### 8.5.4 风险区

展示本章内部的问题：

- 输入不足。
- 内部逻辑不闭合。
- 输出没有推动总轴。
- 人物行动缺少目标驱动。
- 信息流不可信。
- 情绪强度没有事件支撑。

#### 8.5.5 同步建议区

展示本章写完或确认后，哪些正式文档可能需要同步：

- 分章大纲。
- 分章简纲。
- 事件脉络。
- 势力目标线。
- 时间线统计。
- 人物群像或人物梗概。

同步建议不能自动写回正式文档。必须等待用户确认。

### 8.6 新线索准入页面

用于判断是否加入新线索。

必须通过问题链进行判断：

1. 它解决哪个全局结构问题？
2. 它属于已有线索，还是需要独立成线？
3. 它进入哪个阶段？
4. 它需要哪些事件承接？
5. 它需要哪些章节承载？
6. 它改变哪些角色目标？
7. 它改变哪些信息流？
8. 它是否抢走主线？
9. 不加入是否也能完成同样功能？

输出结论：

- 建议加入。
- 建议并入已有线。
- 建议延后。
- 建议废弃。
- 需要先补前置事件。

### 8.7 风险报告页面

按风险类型生成报告。

风险类型：

- 时间矛盾。
- 因果跳跃。
- 目标突变。
- 信息流断裂。
- 章节输入不足。
- 章节输出无效。
- 线索断裂。
- 事件线与人物后台状态混淆。
- 人物条虚段缺少触发或证据。
- 新线索抢主线。
- 支持文档同步冲突。
- 结构事实与章节事实混淆。

## 9. 工作流需求

### 9.1 默认工作流

工具打开后的默认路径：

```text
全局故事地图
-> 当前阶段
-> 当前事件群
-> 活跃线索
-> 章节函数
-> 风险报告
-> 同步建议
```

### 9.2 章节分析工作流

进入章节函数时，必须按这个顺序：

```text
确认全局位置
-> 检查输入
-> 拆内部逻辑
-> 检查输出
-> 生成风险
-> 给出同步建议
```

禁止直接从章节正文开始分析，除非用户明确要求只做文本层面的局部检查。

### 9.3 新线索判断工作流

```text
输入新线索想法
-> 选择它试图解决的问题
-> 挂接全局阶段
-> 挂接已有线索或创建候选线
-> 检查事件承接
-> 检查章节承载
-> 检查目标变化
-> 检查信息流变化
-> 输出准入结论
```

## 10. 状态模型

工具必须区分以下状态。

### 10.1 文本状态

- 未写。
- 规划中。
- 已成文。
- 已修订。

### 10.2 确认状态

- 待确认。
- 已确认。
- 有争议。

### 10.3 同步状态

- 未同步。
- 部分同步。
- 已同步。

### 10.4 风险状态

- 正常。
- 需回看。
- 阻塞。

任何对象不得只用一个“完成”状态概括全部进度。

## 11. 源文件与真相类型

工具读取正式文件时，必须保留文件边界。

建议真相类型：

| 正式文件 | 主要真相类型 |
| --- | --- |
| 分章大纲 | 章节骨架 |
| 分章简纲 | 压缩章节地图 |
| 事件脉络 | 事件与时间推进 |
| 势力目标线 | 目标变化与势力行动 |
| 时间线统计 | 日期锚点与时间矛盾 |
| 人物群像/人物梗概 | 人物功能、关系和改写风险 |
| 正文 | 已成文文本事实 |
| 碎片池 | 候选想法，不自动升格为事实 |

工具不得把这些文件混成一个全能摘要。

## 12. 数据输入与输出

### 12.1 输入类型

工具应支持以下输入：

1. 手工录入的全局阶段。
2. 手工录入或从文档整理出的事件。
3. 手工录入的线索流。
4. 手工录入的章节函数。
5. 从正式文档引用的来源路径。
6. 用户输入的新线索想法。

第一版可以使用样板数据，不要求自动解析全文。

### 12.2 输出类型

工具应输出：

1. 全局故事地图。
2. 阶段结构报告。
3. 事件群报告。
4. 线索流报告。
5. 章节函数报告。
6. 新线索准入报告。
7. 风险报告。
8. 同步建议。

输出默认写入 `验证工具` 内，不得直接覆盖正式文档。

## 13. 小程序原型需求

### 13.1 第一版小程序范围

第一版小程序只做验证性原型。

应包含：

1. 全局故事地图页面。
2. 阶段视图页面。
3. 事件群视图页面。
4. 章节函数页面。
5. 风险报告页面。

可用静态样板数据模拟。

### 13.2 第一版样板数据范围

样板数据可覆盖：

- 皮革厂采访启动。
- 大仙祠显形。
- 慈善晚宴吸收。
- 会后缓冲。
- 群众争夺。
- 流血冲突。
- 敌境陷阱。
- 吕玉接管主线。

其中“第十五章”只能作为“会后缓冲”阶段中的一个章节函数样例，不得作为首页主入口。

### 13.3 交互要求

最低交互要求：

1. 首页能点击阶段。
2. 阶段页能点击事件群。
3. 事件群页能点击章节。
4. 章节页能在输入、内部运作、输出、风险之间切换。
5. 风险报告能返回对应事件、线索或章节。
6. 页面上必须有明确的“你现在处于全局故事流的哪里”的位置提示。

## 14. 验收标准

第一版需求实现后的验收标准：

1. 打开工具后，用户首先看到的是全局故事事件流，而不是某一章。
2. 用户能理解整体故事主轴如何推进。
3. 用户能从全局阶段下钻到事件群。
4. 用户能从事件群下钻到章节函数。
5. 章节函数页面能明确展示输入、内部逻辑、输出、风险、同步建议。
6. 第十五章只作为样例切片出现，不作为工具主入口。
7. 工具不会修改外层正式小说文件。
8. 工具能生成或展示风险报告。
9. 工具能说明哪些内容是规划、哪些已成文、哪些待确认、哪些已同步。
10. 用户看完后能回答：“整个故事现在推进到哪里，下一步要验证什么？”

## 15. 实施阶段建议

### 阶段 P0：需求评审

目标：确认本需求文档的核心模型是否正确。

产物：

- 本需求规格说明。
- 用户评审意见。

验收：

- 用户确认主入口是全局事件流。
- 用户确认章节函数模型成立。
- 用户确认第一版小程序范围。

### 阶段 P1：静态交互原型

目标：建立可点击 HTML 小程序，不接真实数据。

产物：

- 全局故事地图页面。
- 阶段视图页面。
- 事件群视图页面。
- 章节函数页面。
- 风险报告页面。

验收：

- 能按默认工作流从全局走到章节函数。
- 第十五章仅作为样例切片。
- 所有文件在 `验证工具` 内。

### 阶段 P2：样板数据结构化

目标：把样板数据从 HTML 中抽离为结构化数据。

产物：

- 样板 JSON 或 Markdown 数据。
- 数据对象说明。
- 数据校验脚本。

验收：

- 能读取阶段、事件群、事件、线索、章节函数。
- 能发现缺字段、重复 ID 和状态不完整。

### 阶段 P3：报告生成

目标：生成 Markdown 风险报告和同步建议。

产物：

- 风险报告生成器。
- 同步建议报告。

验收：

- 报告只写入 `验证工具`。
- 不直接修改正式小说文档。

## 16. 开放问题

以下问题需要用户后续确认：

1. 全局故事地图是否只覆盖《致命采访》当前卷，还是要覆盖整个改写项目？
2. 阶段编号是否沿用现有 S0-S6，还是重新定义更大的全书阶段？
3. 事件编号是否继续沿用 E001-E048，还是允许按全书重新编号？
4. 第一个交互原型是否只用样板数据，还是读取现有 `事件脉络` 和 `势力目标线` 的部分内容？
5. 章节函数是否需要支持“正文已写”和“尚未写作”两种模式？

## 17. 当前结论

工具的核心不是“章节准备包”，而是“全局故事事件流验证”。

章节是重要对象，但它的角色是全局事件流中的函数节点。章节分析的价值在于拆开黑盒，验证输入是否齐备、内部逻辑是否闭合、输出是否推动总轴。

后续所有原型和实现都应以这个模型为准。
