# 章节事件加工台对象锚点与连续事件条设计

版本：v0.4  
状态：供原型评审  
适用范围：`验证工具` 内的叙事验证工具原型与后续验证性小程序  
关联原型方向：v21 章节事件加工台连续事件条与锚点交互原型  
承接文档：`2026-06-21-章节事件加工台与对象编辑边界设计-v0.3.md`

## 1. 本次修订目的

v20 已经把章节页从全局时间轴中拆出，形成“章前输入 / 中段处理 / 章后输出”的三段式章节加工台。但 v20 暴露了两个对象模型问题：

1. 章节开始时仍在延续的事件被画成左侧一段和中区一段，破坏了“同一事件条”的连续性。
2. 已结束历史事件到新建事件的关系线没有严格连接对象端点，导致连线像装饰箭头，而不是对象关系。

本次修订固化两条核心规则：

```text
延续事件 = 同一条跨列事件条
关系连线 = 对象端点到对象端点
```

## 2. 章节加工台的三类对象

### 2.1 已结束历史事件

已结束历史事件是“来源候选”，不是章节内部正在推进的事件条。

页面表现：

- 位于章前输入列。
- 以短卡片或短条目呈现。
- 右侧必须有关系端点。
- 不能自动进入中段处理区。
- 不能在中段处理区重复画同名对象。

操作规则：

```text
从历史事件右侧端点拖出辅助线
  -> 释放到中段空白行：新建章内事件
  -> 释放到已有章内事件：建立关系
```

### 2.2 章节开始时仍在延续的事件

延续事件是同一条事件，不是“来源卡片 + 中段条”的两份对象。

页面表现：

- 从章前输入列开始。
- 沿同一行水平延展。
- 可以延展到中段处理列，也可以延展到章后输出边界。
- 左侧起点、中间经过段、右侧端点都属于同一个 DOM/对象语义。
- 不允许切成不平行的两段。

操作规则：

```text
拖动延续事件右端点
  -> 停在中段处理：本章只推进到中段
  -> 停在章后输出边界：该事件横跨整章
  -> 继续进入输出区：形成后续输出事件或保留入口
```

### 2.3 章内新建事件

章内新建事件是由一个或多个来源端点拖线生成，或由中段空白行手动创建的事件。

页面表现：

- 位于中段处理区。
- 左侧有输入锚点。
- 右侧有输出/拉伸端点。
- 可以被拉长，表达它在章节内部的处理范围。
- 可以接收多个来源端点，形成多对一。
- 可以输出到多个章后事件，形成一对多。

## 3. 锚点规则

所有关系线必须连接对象锚点。

### 3.1 来源锚点

来源锚点位于对象右侧中线：

- 已结束历史事件：右侧中线。
- 延续事件：右侧当前端点。
- 新建事件：右侧输出端点。

### 3.2 目标锚点

目标锚点位于对象左侧中线：

- 新建事件：左侧输入端点。
- 已存在章内事件：左侧输入端点。
- 章后输出事件：左侧输入端点。

### 3.3 连线约束

连线必须满足：

1. 从左向右。
2. 端点到端点。
3. 不悬空。
4. 不用箭头替代关系对象。
5. 多对一关系应合流到同一个目标锚点。
6. 一对多关系应从同一个来源锚点分裂到多个目标锚点。

## 4. 拖线生成事件流程

推荐交互流程：

```text
按住来源事件右侧端点
  -> 拖出辅助线
  -> 移动到中段处理区的空白行
  -> 辅助线吸附到可创建位置
  -> 释放
  -> 新建一个章内事件
  -> 自动建立 来源事件 -> 新建事件 的关系线
```

如果释放到已有事件的左侧输入锚点：

```text
按住来源事件右侧端点
  -> 拖出辅助线
  -> 吸附到已有事件左侧输入锚点
  -> 释放
  -> 建立关系，不新建事件
```

## 5. 事件条拉伸规则

新建事件和延续事件都可以拉伸，但语义不同：

| 对象 | 拉伸含义 | 允许范围 |
| --- | --- | --- |
| 延续事件 | 调整该事件在本章内持续到哪里 | 从章前输入开始，到中段或章后输出 |
| 新建事件 | 调整该事件在本章内部处理跨度 | 从新建位置开始，到中段或输出边界 |
| 已结束历史事件 | 不允许拉伸，只能拖出关系线 | 章前输入列内部 |

## 6. 与 v0.3 的关系

v0.3 中“三段式章节函数”的结论继续有效：

```text
章前输入 -> 中段处理 -> 章后输出
```

本次 v0.4 只补充三段之间的对象交互语义：

- 不是所有输入都进入中段。
- 延续事件必须是一条连续对象。
- 已结束历史事件通过锚点关系进入中段。
- 新建事件由关系拖拽或中段创建产生。

## 7. 原型验收问题

v21 原型需要验证：

1. 延续事件是否能被一眼看作同一条跨列事件条。
2. 已结束历史事件是否不再和中段新建事件重复。
3. 从历史事件端点到新建事件端点的关系线是否准确、可理解。
4. 从端点拖出辅助线生成新事件的交互是否符合预期。
5. 新建事件和延续事件的拉伸语义是否足够区分。
