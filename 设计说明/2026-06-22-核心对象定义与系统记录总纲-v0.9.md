# 核心对象定义与系统记录总纲 v0.9

## 1. 本文目的

叙事验证工具不能只靠页面流程、按钮和交互原型推进。系统必须先明确自己管理哪些对象，以及这些对象如何被记录、引用和展示。

本文用于定义系统中的核心对象：

1. 章节
2. 事件
3. 剧情线
4. 人物
5. 地点
6. 组织
7. 物件 / 证据 / 材料
8. 对象之间的连接与投影

其中，`2026-06-22-事件定义与系统记录设计-v0.8.md` 是事件对象的细化文档。本文是更上层的对象总纲。

## 2. 系统的基本判断

系统要管理的不是“页面”，而是叙事对象。

页面只是对象的不同视图：

| 页面 | 主对象 | 作用 |
| --- | --- | --- |
| 全局时间画布 | 剧情线、章节、事件 | 查看全局进度、事件跨章分布、剧情线推进。 |
| 章节加工台 | 章节、事件投影 | 处理当前章节如何接收、推进、生成、输出事件。 |
| 事件详情页 | 事件 | 编辑起因、过程、结果和事件关系。 |
| Inspector | 当前选中对象 | 展示摘要、关系和进入详情入口。 |

因此，后续任何界面设计都必须先回答：当前正在操作哪个对象。

## 3. 主对象列表

### 3.1 章节

章节是正文加工单元。

章节不是故事主轴，也不是一个大事件。它的职责是接收若干事件状态，在正文中处理它们，然后输出新的事件状态。

系统中，章节应记录为：

```yaml
chapter:
  id: C15
  title: 私线偏移
  volumeId: V02
  order: 15
  summary: 本章处理会议结果后的私线偏移与后续争夺压力。
  realTimeRange:
    start: 2025-07-16
    end: 2025-07-17
  manuscriptStatus: planning
  inputEventIds:
    - E021
    - E028
  continuingEventIds:
    - E029
  createdEventIds:
    - E041
  outputEventIds:
    - E042
  involved:
    characters:
      - 叶依人
      - 张航远
      - 林倩
    locations:
      - 电视台
    organizations:
      - 电视台
      - 杨家相关方
```

章节对象至少要回答：

1. 本章接收了哪些事件。
2. 本章推进了哪些未结束事件。
3. 本章新建了哪些事件。
4. 本章终结或输出了哪些事件。
5. 本章涉及哪些人物、地点和组织。

章节页面不应变成正文编辑器。它是“章节函数”的操作台。

### 3.2 事件

事件是叙事变化的最小可追踪对象。

事件不是时间段，也不是人物动作本身，而是一个能被记录、引用、连接、推进、终结和输出的变化过程。

事件对象的详细定义见：

`2026-06-22-事件定义与系统记录设计-v0.8.md`

在总纲层面，事件至少包含：

```yaml
event:
  id: E021
  title: 会议结果落地
  summary: 会议决议开始转化为可执行安排。
  type: story_event
  lifecycleStatus: active
  storylineIds:
    - interview
  chapterSpan:
    start: C13
    end: C14
  participants:
    characters:
      - 叶依人
    locations:
      - 电视台
    organizations:
      - 电视台
  cause:
    summary: 前序采访材料和会议压力共同推动。
  process:
    summary: 各方开始把会议结果转化为行动。
  outcome:
    summary: 后续争夺与私线偏移被触发。
```

事件对象至少要回答：

1. 它是什么。
2. 它为什么发生。
3. 它如何发生。
4. 它改变了什么。
5. 它从哪些事件来，又指向哪些事件。

### 3.3 剧情线

剧情线是跨章节追踪一组相关事件的组织方式。

剧情线不是时间轴本身，也不是章节集合。它的作用是把一组有共同主题、对象、因果方向或行动目标的事件串起来。

系统中，剧情线应记录为：

```yaml
storyline:
  id: interview
  title: 采访线
  type: main
  summary: 叶依人、张航远、林倩围绕皮革厂与病患群体展开的采访推进。
  chapterRange:
    start: C10
    end: C18
  eventIds:
    - E010
    - E014
    - E021
    - E029
  primaryCharacterIds:
    - char_yeyiren
    - char_zhanghangyuan
    - char_linqian
  status: active
  visual:
    color: teal
    collapsed: false
```

剧情线对象至少要回答：

1. 这条线追踪什么。
2. 它包含哪些事件。
3. 它横跨哪些章节。
4. 哪些人物在这条线里反复参与。
5. 它当前是推进中、暂停、收束还是已结束。

剧情线不直接记录“发生了什么”。真正发生的是事件。剧情线只是把事件组织成可查看的轨道。

### 3.4 人物

人物是参与事件、推动事件或被事件改变的行动对象。

人物不等于人物线。人物是全局对象；人物线是某个人在某条剧情线里的参与投影。

系统中，人物应记录为：

```yaml
character:
  id: char_yeyiren
  name: 叶依人
  role: 主视角人物
  summary: 由主持人身份进入调查者身份，并在采访、慈善和家族压力之间被迫选择。
  relatedStorylineIds:
    - interview
    - charity
  relatedEventIds:
    - E001
    - E021
    - E029
  status:
    publicIdentity: 主持人
    currentNarrativeRole: 记者 / 调查者
```

人物对象至少要回答：

1. 他是谁。
2. 他在哪些剧情线中出现。
3. 他参与了哪些事件。
4. 他的状态如何被事件改变。

### 3.5 人物线

人物线是“人物 + 剧情线”的组合对象。

它不是人物传记，也不是全局人物详情页。它只说明某个人在某条剧情线里的参与方式。

系统中，人物线应记录为：

```yaml
characterLane:
  id: lane_interview_yeyiren
  characterId: char_yeyiren
  storylineId: interview
  eventIds:
    - E010
    - E014
    - E021
  visibleSegments:
    - start: C10
      end: C15
      mode: explicit
  backgroundSegments:
    - start: C15
      end: C16
      mode: background
```

人物线至少要回答：

1. 这个人物在哪条剧情线里。
2. 哪些段落是正文明确描写。
3. 哪些段落只是后台持续状态。
4. 这个人物在该线中参与了哪些事件。

人物线不能替代事件。人物做了但正文没有呈现的后台行为，可以记录为人物线背景段，但不应强行拉长故事事件条。

### 3.6 地点

地点是事件发生、信息流动或人物行动的空间锚点。

系统中，地点应记录为：

```yaml
location:
  id: loc_tv_station
  name: 电视台
  type: media_space
  summary: 采访任务、素材整理和舆论压力的工作空间。
  relatedEventIds:
    - E010
    - E021
  relatedOrganizationIds:
    - org_tv_station
```

地点至少要回答：

1. 它是什么空间。
2. 哪些事件在这里发生。
3. 它和哪些人物、组织有关。
4. 它是否改变事件的信息流动或行动边界。

### 3.7 组织

组织是事件背后的结构性力量。

组织可以是正式机构，也可以是松散势力、平台、资源网络或利益共同体。

系统中，组织应记录为：

```yaml
organization:
  id: org_tv_station
  name: 电视台
  type: media_organization
  summary: 采访任务的发起和播出平台，同时承受外部压力。
  goalSummary: 控制报道风险并保留新闻价值。
  relatedEventIds:
    - E010
    - E021
  relatedCharacterIds:
    - char_yeyiren
    - char_zhanghangyuan
```

组织至少要回答：

1. 它是什么力量。
2. 它追求什么目标。
3. 它如何影响事件。
4. 它和哪些人物、地点、剧情线相关。

### 3.8 物件 / 证据 / 材料

物件、证据和材料是事件中的信息承载物。

它们不是天然事件，但可以触发事件、支持事件、改变事件判断。

系统中应记录为：

```yaml
evidenceObject:
  id: obj_interview_notes
  title: 采访材料
  type: document
  summary: 采访组整理出的病患、补偿、组织关系材料。
  relatedEventIds:
    - E014
    - E021
  holderCharacterIds:
    - char_linqian
```

物件至少要回答：

1. 它是什么。
2. 谁持有或使用它。
3. 它支持或触发了哪些事件。
4. 它是否改变人物判断或组织行动。

## 4. 连接对象与投影对象

主对象本身不解决所有关系。系统需要额外记录连接和投影。

### 4.1 事件关系

记录事件与事件之间的前后关系。

```yaml
eventRelation:
  id: R021_029
  sourceEventId: E021
  targetEventId: E029
  relationType: causes
  chapterId: C15
  status: confirmed
```

事件链应由事件关系生成，而不是靠画布箭头临时表达。

### 4.2 章节事件投影

记录事件在某一章中的角色。

```yaml
chapterEventProjection:
  id: C15_E021
  chapterId: C15
  eventId: E021
  inputType: continuing_at_chapter_start
  roleInChapter: continued_and_outputs
  rangeInChapter:
    startZone: input
    endZone: output
```

章节加工台主要编辑这一层。

### 4.3 剧情线事件归属

记录事件属于哪条剧情线，以及在该线中的显示方式。

```yaml
storylineEventMembership:
  id: interview_E021
  storylineId: interview
  eventId: E021
  laneRole: main
  orderHint: 21
```

同一个事件可以被多条剧情线引用，但系统必须明确主归属和引用关系。

### 4.4 人物参与记录

记录人物在事件中的作用，而不是只记录“参与人物名单”。

```yaml
characterEventParticipation:
  id: E021_char_yeyiren
  eventId: E021
  characterId: char_yeyiren
  roleInEvent: affected_decision_maker
  actionSummary: 被会议结果推入新的判断压力。
  stateChange: 从材料观察转向行动判断。
```

这能让事件详情页和人物线详情页共享同一套参与记录。

## 5. 对象之间的基本关系

系统关系可以简化为：

```text
剧情线 包含 / 引用 事件
章节 处理 事件
事件 连接 事件
人物 参与 事件
地点 承载 事件
组织 影响 事件
物件 支持 / 触发 事件
人物线 = 人物 在 某条剧情线中的参与投影
```

其中，事件是变化单元；章节是加工单元；剧情线是组织单元。

三者不能互相替代。

## 6. Inspector 的对象读取原则

Inspector 必须先识别当前选中对象类型，再读取对应记录。

| 选中对象 | Inspector 应读取 |
| --- | --- |
| 事件 | 事件主记录、章节投影、前后事件关系、参与对象。 |
| 章节 | 章节主记录、输入事件、延续事件、新建事件、输出事件、涉及对象。 |
| 剧情线 | 剧情线主记录、事件列表、章节范围、主要人物、当前状态。 |
| 人物线 | 人物记录、剧情线记录、人物参与事件、实写段、后台段。 |
| 关系线 | 事件关系记录，包含来源、目标、关系类型和状态。 |

Inspector 不是详情页。它只做：

1. 确认选中对象。
2. 展示必要摘要。
3. 展示关键关系。
4. 提供进入详情或执行明确操作的按钮。

## 7. 页面设计必须服从对象定义

后续原型应按以下顺序设计：

1. 先定义对象。
2. 再定义对象之间的关系。
3. 再定义当前页面操作哪些对象。
4. 最后设计交互与视觉。

不能先画按钮、流程、卡片，再反推系统里应该有什么。

如果某个页面无法说明主对象是什么，就不应继续细化该页面。

## 8. 当前系统的对象边界

### 8.1 全局时间画布

主对象：

- 剧情线
- 章节
- 事件
- 人物线

它不负责编辑事件内部内容，只负责全局定位和对象选择。

### 8.2 章节加工台

主对象：

- 章节
- 章节事件投影
- 事件关系

它不直接编辑事件的起因、过程、结果，只决定事件在当前章节中如何被接入、推进、生成、终结和输出。

### 8.3 事件详情页

主对象：

- 事件
- 事件关系
- 人物参与记录

它负责编辑事件本身的起因、过程、结果、状态变化和前后关系。

### 8.4 人物线页

主对象：

- 人物
- 剧情线
- 人物参与记录
- 实写段 / 后台段

它不替代人物传记，也不替代事件详情页。

## 9. 判断一个设计是否合格

任何新功能或新页面，都必须能回答：

1. 主对象是什么。
2. 它读取哪些记录。
3. 它写回哪些记录。
4. 它是否改变事件链。
5. 它是否改变章节投影。
6. 它是否只是画布状态，而不是业务记录。

如果这六个问题答不上来，说明设计还停留在界面层，不应进入实现。

## 10. 后续补充方向

后续可以继续分别补充：

1. 章节定义与系统记录细化。
2. 剧情线定义与系统记录细化。
3. 人物线定义与系统记录细化。
4. 地点、组织、物件的轻量索引定义。
5. 从现有 `致命采访-事件脉络.md` 到结构化事件对象的迁移规则。

当前优先级应是：

1. 先稳定核心对象定义。
2. 再更新 Inspector。
3. 再更新章节加工台交互。
4. 最后再做更完整的数据编辑。
