# 对象详情页与 Inspector 路由设计

版本：v0.1  
状态：待用户确认  
适用范围：`验证工具` 内的叙事验证工具原型与后续验证性小程序  
关联原型方向：v10 对象详情页路由原型  

## 1. 背景

当前多轨时间线已经能表达真实时间底盘、章节覆盖层、故事线事件、人物子线和人物实虚段。但右侧 Inspector 仍偏向摘要展示，详情仍停留在同页面的小面板里。

用户提出新的核心要求：

1. 故事轴里的每个事件应有基础摘要、参与人物、起因、过程和结果。
2. 进入事件后，应以“事件”为对象，展示事件级详情页，可支持显示或编辑。
3. 章节函数不应继续停在当前画布右侧，而应进入独立页面，明确展示输入、处理、输出。
4. 人物线的详情形态暂不确定，但应先保留对象边界。

## 2. 设计判断

主画布不应承担所有信息。主画布的职责是“全局定位和对象选择”；Inspector 的职责是“确认选中对象并给出进入动作”；详情页的职责是“围绕一个对象展开分析、编辑和验证”。

因此本轮不把更多字段塞进右侧 Inspector，而是建立三层结构：

```text
全局时间线画布 -> Inspector 摘要 -> 对象详情页
```

对象详情页按对象类型区分，不使用一个通用大详情页吞并所有内容。

## 3. 三类对象边界

### 3.1 故事事件

故事事件是故事线上的动作对象。它回答“发生了什么，以及这个发生改变了什么”。

事件详情页必须包含：

- 基础摘要：一句话定义事件。
- 所属故事线：事件挂在哪条故事线。
- 时间与覆盖章节：真实时间范围，以及被哪些章节呈现。
- 参与人物：哪些人物参与，以及每个人在事件中的作用。
- 起因：事件为什么会发生。
- 过程：事件如何推进。
- 结果：事件造成什么状态变化。
- 风险或约束：事件不能被误读或扩写成什么。

事件页可以有编辑形态，但编辑对象仍然是事件本身，不能把章节、人物传记或全局故事线编辑混进同一个界面。

### 3.2 章节

章节是正文函数，不是故事线事件，也不是主轴。它回答“这一章接收什么、内部如何处理、输出什么”。

章节函数页必须包含：

- 输入：进入本章前已经存在的事件状态、人物状态、信息状态、未解决压力。
- 处理：本章内部如何通过场景、冲突、视角、信息释放和节奏安排处理输入。
- 输出：本章结束后，事件线、人物状态、信息暴露和后续风险发生什么变化。

章节函数页可以引用事件和人物线，但它的主对象仍是章节。章节页不应变成事件详情页，也不应变成正文编辑器。

### 3.3 人物线

人物线暂不定义为全局人物传记页。当前更准确的对象是“某个人在某条故事线里的局部参与线”。

人物线 Inspector 应先表达：

- 人物是谁。
- 所属故事线是什么。
- 该人物在这条故事线里的实写段和后台虚段。
- 同一人物还出现在哪些故事线里。
- 当前人物线和故事事件边界之间的风险。

后续若进入人物线详情页，页面主对象应是“人物 / 故事线”的组合，而不是单独人物。

## 4. Inspector 规则

Inspector 是确认层，不是详情页。

选中不同对象后，Inspector 应显示不同摘要：

| 选中对象 | Inspector 摘要重点 | 主动作 |
| --- | --- | --- |
| 故事事件 | 摘要、故事线、章节覆盖、参与人物、起因/过程/结果缩略 | 进入事件页 |
| 章节 | 章节编号、时间覆盖、输入/处理/输出缩略、风险 | 进入章节函数页 |
| 人物线 | 人物、所属故事线、实写段/后台段、跨线出现提示 | 进入人物线页或暂不进入 |

Inspector 中的按钮必须是显式动作。点击对象只选中，不自动进入详情。

## 5. 路由模型

原型阶段建议使用单页路由模拟多页面，避免过早拆分复杂文件。

推荐路由：

```text
#timeline
#event/<eventId>
#chapter/<chapterId>
#lane/<trackId>/<characterId>
```

其中：

- `#timeline`：全局画布。
- `#event/e-main-13`：事件详情页。
- `#chapter/C13`：章节函数页。
- `#lane/interview/yushu`：人物线详情页。

后续实现时可以转成真实前端路由，但对象边界不应变化。

## 6. 页面结构建议

### 6.1 事件详情页

事件详情页建议分为四个区域：

1. 顶部对象栏：事件标题、摘要、故事线、时间范围、覆盖章节。
2. 因果拆解：起因、过程、结果。
3. 参与人物：每个人的事件作用、立场、状态变化。
4. 编辑或检查区：基础字段、风险、关联事件、返回全局画布。

### 6.2 章节函数页

章节函数页建议分为四个区域：

1. 顶部对象栏：章节编号、标题、时间范围、正文阶段。
2. 三段函数：输入、处理、输出。
3. 章节内部对象：本章涉及的事件、人物状态、信息变化。
4. 检查区：章节是否闭合、是否承接上游、是否制造下游。

### 6.3 人物线页

人物线页建议先做低承诺原型：

1. 顶部对象栏：人物 / 故事线。
2. 参与段列表：实写段、后台虚段。
3. 跨线出现：同一人物出现在其他故事线的提示。
4. 风险检查：人物后台状态是否被误写成事件线延长。

## 7. 数据字段建议

### 7.1 事件字段

```text
id
title
summary
trackId
startAt
endAt
chapterRefs
participants[]
cause
process
result
risk
status
```

`participants[]` 建议包含：

```text
characterId
role
action
stateChange
visibility
```

### 7.2 章节字段

```text
id
title
startAt
endAt
input
process
output
eventRefs[]
characterStateRefs[]
risk
```

### 7.3 人物线字段

```text
trackId
characterId
summary
explicitSegments[]
backgroundSegments[]
relatedEvents[]
crossTrackRefs[]
risk
```

## 8. 非目标

本设计暂不解决：

1. 正式数据库结构。
2. 多人协作和权限。
3. 正文编辑器。
4. 人物全局传记页。
5. 复杂图数据库关系推理。
6. 从小说正文自动抽取全部事件。

## 9. 验收问题

本轮原型图需要用户判断：

1. 事件页是否足以承载事件级编辑和检查。
2. 章节函数页是否符合“输入、处理、输出”的黑盒定义。
3. 人物线页是否应该继续作为组合对象，而不是全局人物页。
4. Inspector 是否应只做摘要和入口，不再承载完整详情。
