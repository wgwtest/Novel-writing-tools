# 事件定义与系统记录设计 v0.8

## 1. 本文目的

本文用于明确“事件”在叙事验证工具中的系统记录方式。

前几版原型已经验证了全局时间画布、章节加工台、右侧 Inspector、延续事件条、关系线编辑等界面形态，但这些界面仍缺少一个稳定基础：系统到底如何记录“事件”这个对象。

如果事件对象没有定义清楚，Inspector 就只能显示临时说明；关系线也只是画面上的线；章节加工台里的输入、处理、输出都会变成界面摆放，而不是可追踪的叙事结构。

因此，本轮先定义事件对象在系统中的记录方式。后续原型、Inspector、事件详情页和章节工作台都应以本文为依据。

## 2. 事件的系统定义

事件是小说叙事中的独立变化单元。

在系统中，一个事件不是单纯的时间段，也不是单个人物动作，而是一个可被记录、引用、连接、推进、终结和输出的叙事对象。

一个事件至少回答四个问题：

1. 它是什么。
2. 它为什么发生。
3. 它如何发生。
4. 它改变了什么。

因此，事件记录必须同时包含身份、位置、参与对象、起因、过程、结果和关系。

## 3. 事件记录的三层结构

系统中不应只用一张“事件卡片”记录所有内容。事件应拆成三层：

1. 事件主记录
   - 记录事件自身是什么。
   - 这是事件对象的主数据。

2. 事件关系记录
   - 记录事件之间怎么连接。
   - 多对一、一对多、延续、转化、触发，都通过关系记录表达。

3. 章节投影记录
   - 记录某个事件在某一章里如何出现、被处理、被输出。
   - 同一个事件可以在不同章节中有不同投影，但不能被复制成多个互不关联的新事件。

这三层分开后，画布、章节工作台和 Inspector 才能各自读取自己需要的内容。

## 4. 事件主记录

事件主记录是系统里的事件本体。

建议字段如下：

```yaml
event:
  id: E021
  title: 会议结果落地
  summary: 会议决议开始转化为可执行安排，采访线进入新的压力阶段。
  type: story_event
  lifecycleStatus: active
  storylineIds:
    - interview
  chapterSpan:
    start: C13
    end: C14
  realTimeRange:
    start: 2025-07-13
    end: 2025-07-14
  participants:
    characters:
      - 叶依人
      - 张航远
      - 林倩
    locations:
      - 电视台
    organizations:
      - 电视台
      - 杨家相关方
    objects:
      - 采访材料
  cause:
    summary: 前序采访材料和会议压力共同推动决议落地。
  process:
    summary: 各方开始把会议结果转化为具体动作和限制。
  outcome:
    summary: 后续争夺和私线偏移被触发。
    stateChanges:
      - 采访组从材料整理转入执行压力。
      - 张航远个人判断风险上升。
  notes: ''
```

### 4.1 身份字段

| 字段 | 含义 |
| --- | --- |
| `id` | 事件唯一编号，例如 E021。 |
| `title` | 事件名称，用于画布、列表和 Inspector。 |
| `summary` | 一句话摘要，说明这个事件改变了什么。 |
| `type` | 事件类型，例如故事事件、人物事件、信息事件、组织事件、证据事件。 |
| `lifecycleStatus` | 事件当前生命周期状态，例如未开始、进行中、已结束、暂停、背景持续。 |

### 4.2 位置字段

| 字段 | 含义 |
| --- | --- |
| `storylineIds` | 事件属于哪些故事线。一个事件可以挂在一条主线，也可以被多个故事线引用。 |
| `chapterSpan` | 事件覆盖的章节范围。 |
| `realTimeRange` | 事件对应的真实时间范围。没有精确日期时，可以先记录粗略范围。 |

章节和时间不是事件本身，只是事件的定位方式。

### 4.3 参与对象字段

| 字段 | 含义 |
| --- | --- |
| `characters` | 参与人物。 |
| `locations` | 发生地点或关键空间。 |
| `organizations` | 涉及组织、势力或机构。 |
| `objects` | 关键物件、证据、材料、文件。 |

任何具体事件点都应尽量落到人物、地点、组织和关键物件上。否则事件容易变成空泛判断。

### 4.4 起因、过程、结果字段

| 字段 | 含义 |
| --- | --- |
| `cause` | 事件为什么发生。可以引用前置事件，也可以说明触发条件。 |
| `process` | 事件如何发生。记录人物动作、冲突机制、信息流动和关键转折。 |
| `outcome` | 事件造成什么变化。应记录状态变化，并尽量指向后续事件。 |

这里的起因、过程、结果是事件详情页的主内容。Inspector 只显示缩略摘要。

## 5. 事件关系记录

事件之间的连接不应只存在于画布线条中，而应作为独立记录保存。

建议字段如下：

```yaml
eventRelation:
  id: R021_029
  sourceEventId: E021
  targetEventId: E029
  relationType: causes
  chapterId: C15
  status: confirmed
  summary: 会议结果落地促成会后转争夺。
```

### 5.1 关系字段

| 字段 | 含义 |
| --- | --- |
| `id` | 关系记录唯一编号。 |
| `sourceEventId` | 来源事件。 |
| `targetEventId` | 目标事件。 |
| `relationType` | 关系类型。 |
| `chapterId` | 这条关系在哪个章节加工台中被建立或确认。 |
| `status` | 关系状态，例如候选、已确认、已删除、待复核。 |
| `summary` | 关系说明。 |

### 5.2 关系类型

| 类型 | 含义 |
| --- | --- |
| `causes` | 来源事件直接促成目标事件。 |
| `enables` | 来源事件为目标事件提供条件。 |
| `continues` | 目标事件是来源事件的延续。 |
| `transforms` | 来源事件在章节处理中转化为目标事件。 |
| `blocks` | 来源事件阻碍或压制目标事件。 |
| `reveals` | 来源事件揭示出目标事件或隐藏信息。 |
| `supports` | 来源事件为目标事件提供证据、材料或解释。 |

事件链由关系记录生成，而不是由事件卡片里手写箭头生成。

## 6. 章节投影记录

章节加工台不是直接编辑事件本体的全部内容，而是编辑“事件在当前章节里的投影”。

同一个事件在全局系统中只有一个主记录，但它在某一章中可能表现为：

- 章前已结束历史事件。
- 章前仍在延续的事件。
- 章内新建事件。
- 章内被推进的事件。
- 章内终结的事件。
- 章后输出事件。

这些状态应记录在章节投影中。

建议字段如下：

```yaml
chapterEventProjection:
  id: C15_E021
  chapterId: C15
  eventId: E021
  inputType: continuing_at_chapter_start
  roleInChapter: continued_and_outputs
  rangeInChapter:
    startZone: input
    endZone: output
  laneId: interview
  rowOrder: 3
  displayState:
    selected: false
    collapsed: false
  chapterDecision:
    summary: 本章让该事件横跨整章，并输出为后续争夺事件。
```

### 6.1 输入类型

| 类型 | 含义 |
| --- | --- |
| `completed_before_chapter` | 章节开始前已经结束，只能作为来源或原因。 |
| `continuing_at_chapter_start` | 章节开始时仍在延续，可以被拖入中段或拖到章后输出。 |
| `created_in_chapter` | 在本章中新建。 |

### 6.2 章节角色

| 类型 | 含义 |
| --- | --- |
| `source_only` | 只作为来源，不进入章内处理。 |
| `continued` | 在本章继续推进。 |
| `transformed` | 在本章被转化为新事件。 |
| `completed` | 在本章终结。 |
| `outputs` | 作为章后输出，进入后续章节或全局事件流。 |
| `continued_and_outputs` | 横跨本章，并作为后续输入继续存在。 |

### 6.3 章节范围

| 字段 | 含义 |
| --- | --- |
| `startZone` | 事件条在章节加工台中的起始区域：input、process、output。 |
| `endZone` | 事件条在章节加工台中的结束区域：input、process、output。 |

这能支持用户提出的操作：

- 延续事件可以缩回到章前输入。
- 延续事件可以停在中段处理。
- 延续事件可以横跨整章进入章后输出。
- 已结束历史事件不进入中段，只通过端点关系参与新事件生成。

## 7. 画布记录与业务记录分离

事件对象不应保存具体像素坐标作为主数据。

系统应区分：

1. 业务记录
   - 事件是什么。
   - 事件和谁有关。
   - 事件从哪里来，到哪里去。
   - 事件在本章中是什么角色。

2. 画布显示记录
   - 第几行。
   - 是否选中。
   - 是否折叠。
   - 当前拖拽状态。
   - 临时连线状态。

画布可以保存必要的视图状态，但这些状态不能替代事件定义。

## 8. Inspector 的读取规则

选中事件时，右侧 Inspector 不应显示临时说明，而应读取三层记录：

1. 从事件主记录读取：
   - 事件 ID。
   - 事件名称。
   - 摘要。
   - 生命周期状态。
   - 所属故事线。
   - 参与人物、地点、组织、关键物件。

2. 从章节投影记录读取：
   - 当前章节。
   - 输入类型。
   - 本章角色。
   - 当前范围：章前、处理中、横跨整章、章后输出。
   - 本章处理决定。

3. 从事件关系记录读取：
   - 前置事件。
   - 后置事件。
   - 关系类型。
   - 关系状态。

Inspector 最低展示内容：

| 区块 | 内容 |
| --- | --- |
| 基础信息 | 事件 ID、名称、生命周期、故事线。 |
| 参与对象 | 人物、地点、组织、关键物件。 |
| 当前章节投影 | 输入类型、本章角色、范围、处理决定。 |
| 事件链 | 前置事件、后置事件、关系类型。 |
| 操作 | 进入事件详情、调整范围、删除关系或检查关系。 |

Inspector 是确认层，不是事件详情页。它必须提供“进入事件详情”的显式按钮。

## 9. 事件详情页的读取规则

事件详情页读取事件主记录和完整关系记录。

页面应围绕四个区域展开：

1. 起因
   - 前置事件。
   - 触发条件。
   - 相关人物目标或组织压力。

2. 过程
   - 人物动作。
   - 冲突机制。
   - 信息流动。
   - 关键转折。

3. 结果
   - 状态变化。
   - 后置事件。
   - 被终结、被延续或被转化的对象。

4. 编辑检查
   - 事件边界是否清楚。
   - 是否误把背景状态写成事件。
   - 是否缺少人物、地点或组织承载。
   - 是否有结果但没有前置逻辑。
   - 是否有起因但没有输出变化。

## 10. 与现有事件脉络文件的对应关系

现有 `致命采访-事件脉络.md` 已经有事件对象雏形。

可对应为：

| 现有字段 | 系统记录字段 |
| --- | --- |
| 事件编号，例如 E001 | `event.id` |
| 事件标题 | `event.title` |
| 发生位置 | `participants.locations` 或 `realTimeRange` / `chapterSpan` 的补充说明 |
| 参与人物 | `participants.characters` |
| 主要经过 | `process.summary` |
| 结果变化 | `outcome.summary` 与 `outcome.stateChanges` |

也就是说，当前项目并不是没有事件，而是已经有事件记录草稿。系统需要把这些草稿拆成可引用、可连接、可投影的结构化对象。

## 11. 当前原型的调整依据

基于本文，后续 V25 或后续版本应调整：

1. 右侧 Inspector 增加事件 ID、参与人物、地点、组织、关键物件。
2. 选中事件条时，Inspector 显示事件主记录和当前章节投影。
3. 选中关系线时，Inspector 显示关系记录。
4. 延续事件条不再被视为多段 UI 片段，而是同一个事件的章节投影。
5. 已结束历史事件不进入中段处理，只作为来源事件通过关系记录连接到新事件。
6. 事件详情页负责编辑起因、过程、结果和状态变化。

## 12. 判断标准

一个事件对象是否记录合格，至少满足：

1. 有唯一 ID。
2. 有一句话摘要。
3. 有参与人物、地点或组织中的至少一类承载对象。
4. 有起因、过程、结果的基本描述。
5. 能说明它在当前章节中的角色。
6. 能说明它与前置或后置事件的关系；基础事件可以没有前置事件，但必须有触发条件或背景来源。

如果不能满足这些条件，它可能还不是事件，而是线索、人物状态、背景任务、主题判断或普通说明。
