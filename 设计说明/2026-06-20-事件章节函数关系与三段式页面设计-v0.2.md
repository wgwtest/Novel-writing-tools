# 事件、章节函数关系与三段式页面设计

版本：v0.2  
状态：供原型评审  
适用范围：`验证工具` 内的叙事验证工具原型与后续验证性小程序  
关联原型方向：v11 事件章节三段式关系原型  

## 1. 本次修订目的

v0.1 已经把全局画布、Inspector、对象详情页分开，但事件页仍偏向普通详情表，未充分表达事件与章节之间的处理关系。

本次修订把核心模型明确为：

```text
事件图：前置事件 -> 当前事件处理 -> 后置事件
章节函数：输出事件集 = 章节函数(输入事件集)
```

因此，事件页和章节页都应采用同一种横向工作语法：

```text
左侧输入 -> 中部处理逻辑与处理内容 -> 右侧输出
```

其中，中部是页面主体，也就是用户所说的“猪肚”：它承载真正的处理过程、判断、改写、状态变化和编辑检查。

## 2. 事件的定义

事件是小说叙事中的基础单元。它不是章节的附属标签，而是一个可以被追踪、引用、输入和输出的独立叙事过程。

一个事件回答：

```text
为什么发生 -> 具体发生了什么 -> 造成了什么变化
```

### 2.1 事件的三段结构

| 位置 | 含义 | 页面表现 |
| --- | --- | --- |
| 输入 | 前置事件或前置信息。基础事件可以没有输入。 | 左侧输入列 |
| 处理 | 当前事件内部发生的动作、判断、冲突、人物状态变化。 | 中部主处理区 |
| 输出 | 被当前事件制造、补充、改变或终结的后置事件。 | 右侧输出列 |

事件详情页不应只写摘要。它至少要表达：

- 起因：由哪些前置事件或前置信息触发。
- 过程：谁在什么地点、什么组织压力下，以什么动作推动了事件。
- 结果：输出了哪些新的事件，或让哪些旧事件进入新状态。
- 编辑检查：当前事件是否有明确边界，是否误把人物后台行为写成故事线事件。

### 2.2 起因与结果必须是可追踪对象

起因不应只是散文式说明。它应该尽量指向一个或多个前置事件。

结果也不应只是状态描述。它应该尽量指向一个或多个后置事件。

这意味着事件详情页中的输入卡和输出卡应可点击，后续实现时可以跳转到对应事件页。

### 2.3 基础事件

有些事件是基础事件，没有明确前置事件，例如“发现噪点”或“采访启动”。这类事件的输入可以为空，但仍需写明它的触发条件和叙事位置。

基础事件不是没有逻辑，而是它的逻辑来自世界状态、人物目标、组织结构或前文背景，而不是来自一个明确的前置事件。

## 3. 章节的定义

章节是正文函数。章节不是故事主轴，也不是一个大事件；章节的工作是接收若干事件状态，在正文中处理它们，然后输出新的事件状态。

推荐表达式：

```text
输出事件集 = 章节函数(输入事件集)
```

### 3.1 章节处理的单位是事件

每一章的输入与输出单元应尽量是事件：

- 输入：进入本章前已经存在的事件、未闭合事件、人物状态事件、信息压力事件。
- 处理：本章如何承接、演化、补充、重新包装、创建、终结这些事件。
- 输出：本章结束后形成的新事件、被改写的旧事件、被终结或暂时悬置的事件。

章节函数页的核心不是“这一章写了什么”，而是“这一章如何处理事件”。

### 3.2 章节中的事件操作

章节中常见的事件操作包括：

| 操作 | 定义 |
| --- | --- |
| 承接 | 把上游事件带入本章，并保持其问题未关闭。 |
| 演化 | 让事件从一种状态转成另一种状态。 |
| 补充 | 增加事件的证据、人物行动或信息条件。 |
| 重新包装 | 让同一事实在新的叙事口径中出现。 |
| 创建 | 在本章中制造新的事件节点。 |
| 终结 | 让事件在当前章内闭合，或停止继续推动。 |
| 分裂 | 一个输入事件分出多个下游事件。 |
| 合流 | 多个输入事件汇成一个下游事件。 |

## 4. 事件与章节的关系

事件与章节不是上下级关系，而是处理关系。

```text
前置事件 A/B/C
      ↓
章节 C13 的正文处理
      ↓
后置事件 D/E/F
```

一个事件可以跨越多个章节，一个章节也可以处理多个事件。章节负责正文组织，事件负责因果追踪。

这能避免两个问题：

1. 把章节误当成故事主轴，导致只能看单章走向。
2. 把事件误当成章节摘要，导致事件之间的因果关系断裂。

## 5. 时间点的组成

任何一个具体事件发生点，都不只是时间坐标。它一定由以下三类对象共同构成：

- 人物：谁参与、谁推动、谁被影响。
- 地理：事件发生或被感知的位置。
- 组织：电视台、祠庙解释权网络、慈善资源、平台、群众等结构性力量。

因此，事件页与章节页都应在页面下缘保留“人物 / 地理 / 组织”上下文带。

这个上下文带不替代中部处理区。它的作用是说明当前事件或章节处理发生在哪些对象之间。

## 6. 页面设计规则

### 6.1 事件详情页

事件详情页采用左中右结构：

```text
左：输入事件 / 前置信息
中：当前事件处理逻辑与处理内容
右：输出事件 / 后置逻辑
下：人物、地理、组织上下文
```

中部应最大，因为事件页的核心不是列出字段，而是解释这个事件如何发生并如何造成变化。

### 6.2 章节函数页

章节函数页采用同样结构：

```text
左：输入事件集
中：章节内部如何承接、演化、补充、包装、创建、终结
右：输出事件集
下：本章涉及的人物、地理、组织上下文
```

章节页不应变成正文编辑器，也不应只做章节摘要。它要把章节这个黑盒拆成事件处理台。

### 6.3 Inspector

Inspector 仍是确认层，不是详情页。

选中对象后，Inspector 只做三件事：

1. 确认当前选中的对象类型和对象名。
2. 提供最小摘要，帮助判断是否选对。
3. 提供明确按钮进入事件页、章节函数页或人物线页。

点击画布对象不应自动进入详情页。

## 7. 人物条与故事事件的边界

故事事件只标记文章中发生、呈现或明确推动的事件。

人物条可以表达人物在对应故事线中的参与段，包括：

- 实写段：文章中明确呈现的人物行动。
- 后台段：人物确实在做，但正文没有直接描写的背景状态。

后台段不应把故事事件条拖长。故事事件条只表达正文中发生并承担叙事推进作用的部分。

## 8. 原型验收问题

v11 原型需要评审以下问题：

1. 事件页改成左输入、中处理、右输出后，是否更符合事件的因果定义。
2. 章节函数页是否能清楚表达“章节处理事件集”。
3. 页面下缘的人物、地理、组织上下文是否足以承接具体时间点的组成。
4. 全局画布到详情页的跳转是否仍保持“先选中，再确认进入”的操作原则。
