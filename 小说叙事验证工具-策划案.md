# 小说叙事验证工具策划案

版本：v0.1 策划稿  
范围：仅限 `验证工具` 文件夹内的实验性文档与后续脚本  
当前状态：只做方案说明，不实现代码，不修改外层正式文档

## 1. 工具定位

这个工具不是写作软件，也不是自动续写工具。它的定位是“叙事结构验证与索引工具”。

它要解决的问题是：当前小说项目里已经有正文、分章大纲、事件脉络、势力目标线、人物群像、时间线统计、碎片池等多种材料，但这些材料各自承担不同真相，阅读时需要人工在多个文档之间切换。工具的任务是把这些材料转成可检查、可追踪、可回收的结构视图。

核心目标：

1. 让时间线、事件线、目标线能够互相跳转。
2. 让每个事件的章节功能、信息变化、目标变化、后续钩子清楚可见。
3. 让“已成文”“规划中”“待确认”“已同步”这些状态分开记录。
4. 帮助判断新线索是否有必要加入，是否会压垮主线。
5. 帮助发现时间冲突、因果跳跃、目标突变、信息流断裂、人物行动不合理等问题。

## 2. 隔离原则

所有实验内容只放在 `验证工具` 文件夹内。

工具可以读取或引用外层正式文件，但不能直接覆盖它们。任何检查结论只能先生成报告或建议，由人工确认后再决定是否同步到正式文档。

外层正式文件包括但不限于：

- `致命采访-事件脉络.md`
- `致命采访-势力目标线.md`
- `时间线统计-2025夏季基准.md`
- `致命采访-分章大纲.txt`
- `致命采访-分章简纲.txt`
- `人物梗概.md`
- 各章节正文

## 3. 设计原则

### 3.1 轻结构优先

第一阶段不做复杂数据库，不做可视化大屏，不做编辑器插件。先用 Markdown 表格、事件卡、索引表和检查报告验证方法是否有效。

### 3.2 事件卡为中心

时间线、事件线、目标线都围绕“事件卡”展开。每个事件卡是一个叙事节点，它回答：

- 什么时候发生；
- 哪些人物参与；
- 属于哪些线；
- 改变了什么信息；
- 改变了谁的目标；
- 推动了什么后续事件；
- 当前是否已成文、已确认、已同步。

### 3.3 事件线与人物条分层

故事由事件组成，但事件需要挂在时间根轴上观察。多轨时间轴里必须区分两类东西：

- 事件线：只标记正文中实际发生、实际叙述到的事件段。
- 人物条：标记某个人物在某条事件线下的参与片段。

事件线不是后台状态容器。某个人物在逻辑上还在执行、等待、铺垫，不能因此把事件线事件块拖长。事件线可以由多个离散事件块组成，事件块的边界以“文章是否写到这个事件”为准。

人物条允许更细的表达：

- 实段：正文直接描写、读者明确看到的人物行为、参与或影响。
- 虚段：人物在逻辑上持续执行、维持状态或暗中推动，但正文没有直接写出的后台状态。

虚段必须依附人物条，不能替代事件线事件块。也就是说，“人物做了但文中没写”可以在人物条上用虚段解释，但不能让故事线看起来像正文已经发生了新的事件。

### 3.4 状态分离

不能用一个“完成”表示所有事情。至少要分开：

- 文本状态：未写、规划中、已成文、已修订。
- 确认状态：待确认、已确认、有争议。
- 同步状态：未同步、部分同步、已同步。
- 风险状态：正常、需回看、阻塞。

### 3.5 正式文档不被工具吞并

工具只做索引和验证，不取代原有文档。正式文档仍然保留各自职责：

- 分章大纲：章节骨架。
- 分章简纲：压缩章节地图。
- 事件脉络：按事件记录发生与推进。
- 势力目标线：记录目标变化。
- 时间线统计：记录日期锚点与矛盾。
- 人物群像/人物梗概：记录人物功能、关系和改写风险。

## 4. 预期使用场景

### 4.1 写新章前

输入：目标章节或事件范围，例如“第十五章”或“E029-E035”。

输出：

- 本章必须承接的前置事件；
- 当前主要事件线；
- 当前主要人物目标；
- 不能提前解决的问题；
- 需要避免的时间和信息冲突；
- 本章结束后应留下的钩子。

### 4.2 写完一章后

输入：新章节正文或章节摘要。

输出：

- 新增事件；
- 被改写的目标；
- 需要同步的正式文档；
- 可能产生的新矛盾；
- 是否有事件线断裂或目标跳变。

### 4.3 判断是否增加新线索

输入：一个新想法，例如“增加化学检测线”“增加记者内部背刺线”。

输出：

- 它属于已有哪条线；
- 是否需要独立成线；
- 它服务哪一阶段；
- 它是否有事件承接点；
- 它是否会抢走主线；
- 如果不加入，会损失什么；
- 如果加入，需要删除或延后什么。

### 4.4 检查人物目标是否合理

输入：人物或势力名称。

输出：

- 当前注意力点；
- 总体目标；
- 当前目标；
- 最近一次目标变化由哪个事件触发；
- 下一步行动是否符合目标；
- 是否存在“事件没有发生，但人物突然改变立场”的问题。

### 4.5 检查信息流

输入：某条关键信息，例如“职业病不只是诊断问题，而是材料证据链问题”。

输出：

- 谁知道；
- 谁不知道；
- 谁误解；
- 信息通过什么路径流动；
- 哪个章节让信息进入读者视野；
- 是否存在人物凭空知道或读者被过早告知的问题。

## 5. 功能清单

### F01 源文件登记

目的：明确工具读取哪些正式文档，以及每个文档承担什么真相类型。

做法：

- 建立源文件登记表。
- 记录文件路径、文件职责、可信层级、是否允许自动读取、是否允许建议同步。
- 对每个源文件标注“真相类型”，例如时间真相、事件真相、目标真相、人物功能真相、正文真相。

示例字段：

| 字段 | 说明 |
| --- | --- |
| source_id | 源文件编号 |
| path | 文件路径 |
| truth_type | 真相类型 |
| role | 文件职责 |
| read_mode | 手动读取或脚本读取 |
| sync_policy | 只读、可建议同步、禁止同步 |

### F02 事件卡管理

目的：把每个关键事件变成可追踪对象。

做法：

- 为每个事件建立事件卡。
- 沿用现有 E001、E002 这类事件 ID。
- 每张卡同时挂接时间、章节、事件线、目标线、信息变化和风险。

事件卡建议字段：

| 字段 | 说明 |
| --- | --- |
| event_id | 事件编号 |
| title | 事件名称 |
| status | 已成文、规划中、待确认 |
| time_ref | 日期或相对时间 |
| chapter_ref | 关联章节 |
| location | 地点 |
| participants | 参与人物 |
| event_lines | 所属事件线 |
| goal_lines | 影响的目标线 |
| info_change | 信息变化 |
| goal_change | 目标变化 |
| result_change | 结果变化 |
| next_hooks | 后续钩子 |
| risks | 风险备注 |
| source_refs | 来源文件 |

### F03 时间线视图

目的：检查事件是否在合理时间内发生。

做法：

- 把事件卡按日期或相对日排列。
- 支持绝对日期和相对日期并存。
- 对没有日期的事件允许标记为“锚点后第几天”。
- 标记时间挤压、同一人物多地出现、事件顺序冲突。

检查项：

- 日期是否与已确认锚点冲突。
- 章节内时间跨度是否过长或过短。
- 同一天事件是否超过人物行动容量。
- “几天后”“第二天”“第四天”是否能互相对齐。

### F04 事件线视图

目的：检查每条事件线是否有持续推进。

做法：

- 沿用当前 A-I 事件线索引。
- 为每条线记录当前状态、最近事件、下一事件、断裂风险。
- 支持一个事件挂接多条事件线。
- 事件线上的块只表示正文可见事件，不表示人物后台状态。
- 同一条事件线允许出现多个离散事件块，中间空白不自动等于断裂，需要结合下一期待事件判断。
- 当某个人物在事件外仍有逻辑动作时，放入该线展开后的对应人物条，而不是拉长事件块。

建议字段：

| 字段 | 说明 |
| --- | --- |
| line_id | 线索编号 |
| line_name | 线索名称 |
| current_state | 当前状态 |
| last_event | 最近事件 |
| next_expected_event | 下一步应有事件 |
| visible_segments | 正文可见的事件块 |
| pressure | 这条线当前制造的叙事压力 |
| risk | 断裂、抢戏、重复、提前解决等风险 |

### F04-1 人物条视图

目的：检查某个人物在某条事件线下到底做了什么，以及哪些只是后台状态。

做法：

- 人物条必须挂在某条事件线下，不直接替代事件线。
- 同一人物可以出现在多条事件线下，但每条线下的语义独立。
- 人物条中的片段分为实段和虚段。
- 实段对应正文直接出现的行为、对话、判断、被影响状态。
- 虚段对应正文未直接描写但逻辑上存在的行动、等待、监控、暗中推动或状态维持。
- 虚段必须有触发事件或来源依据，不能作为“想当然存在”的万能解释。

建议字段：

| 字段 | 说明 |
| --- | --- |
| lane_id | 人物条编号 |
| actor_id | 人物编号 |
| parent_line_id | 所属事件线 |
| explicit_segments | 实写片段 |
| background_segments | 后台虚段 |
| trigger_event | 触发该人物状态的事件 |
| source_refs | 来源文件或章节 |
| risk | 虚段缺证据、实段越界、人物功能混淆等风险 |

### F05 目标线视图

目的：检查人物或势力的行动是否由目标驱动。

做法：

- 为每个核心人物或势力建立目标卡。
- 目标卡不复述剧情，只记录目标变化。
- 目标变化必须绑定事件输入。

目标卡建议字段：

| 字段 | 说明 |
| --- | --- |
| actor_id | 人物或势力编号 |
| actor_name | 人物或势力名称 |
| overall_goal | 总体目标 |
| attention | 当前注意力点 |
| current_goal | 当前目标 |
| last_trigger_event | 最近触发事件 |
| action_output | 行动输出 |
| side_effect | 副作用 |
| next_pressure | 下一步压力 |

### F06 状态与同步面板

目的：避免“写了”“确认了”“同步了”混在一起。

做法：

- 为章节、事件、目标变化分别记录状态。
- 每次检查报告都说明是否需要同步正式文档。
- 不把规划中内容自动升格为已确认事实。

状态字段：

| 字段 | 可选值 |
| --- | --- |
| text_state | 未写、规划中、已成文、已修订 |
| confirm_state | 待确认、已确认、有争议 |
| sync_state | 未同步、部分同步、已同步 |
| risk_state | 正常、需回看、阻塞 |

### F07 信息流检查

目的：检查信息是否按可信路径流动。

做法：

- 对关键事实建立信息卡。
- 记录谁知道、谁误解、谁隐瞒、读者何时知道。
- 检查人物是否凭空掌握信息。

信息卡建议字段：

| 字段 | 说明 |
| --- | --- |
| info_id | 信息编号 |
| content | 信息内容 |
| known_by | 已知人物 |
| misunderstood_by | 误解人物 |
| hidden_from | 尚未知情人物 |
| reveal_event | 揭示事件 |
| reader_visible_at | 读者可见章节 |
| credibility_path | 信息可信路径 |

### F08 风险报告

目的：生成具体可处理的问题清单。

风险分类：

- 时间矛盾。
- 事件线断裂。
- 目标突变。
- 信息流断裂。
- 人物行动容量过载。
- 章节功能重复。
- 新线索抢走主线。
- 现实制度或场景逻辑不可信。
- 证据链被万能角色提前解决。
- 情绪强度没有事件支撑。

报告格式：

| 风险等级 | 类型 | 位置 | 问题 | 建议处理 |
| --- | --- | --- | --- | --- |
| 高 | 目标突变 | E041 | 张航远转向过快 | 增加 E033-E039 的连续诱导证据 |

### F09 章节准备包

目的：写某章前快速生成最小上下文。

做法：

- 输入章节号或事件范围。
- 输出本章需要读取的事件卡、目标卡、时间锚点和风险。
- 明确哪些远场内容不需要读。

准备包内容：

- 本章前置状态。
- 本章必须推进的事件线。
- 本章必须改变的目标线。
- 本章不能解决的问题。
- 本章应留下的后续钩子。
- 本章写完后需要检查的文档。

### F10 新线索准入检查

目的：判断一个新线索是否该加入。

准入问题：

1. 它解决哪个现有结构问题？
2. 它是否已有事件承接口？
3. 它是独立事件线，还是某条线的子线？
4. 它会改变谁的目标？
5. 它需要占用几个章节？
6. 它是否会抢走当前主线？
7. 它不加入时是否仍能完成同样功能？

输出结论：

- 建议加入。
- 建议并入已有线。
- 建议延后。
- 建议废弃。
- 需要先补前置事件。

## 6. 数据对象设计

### 6.1 核心对象

| 对象 | 作用 |
| --- | --- |
| Event | 事件卡，工具中心对象 |
| EventLine | 事件线，例如采访主线、大仙祠线、化学证据线 |
| CharacterLane | 人物条，某个人物在某条事件线下的参与视图 |
| CharacterSegment | 人物条片段，分为正文实写段和后台虚段 |
| GoalLine | 人物或势力目标线 |
| TimePoint | 时间点或时间段 |
| Chapter | 章节对象 |
| InfoNode | 关键信息对象 |
| RiskItem | 风险项 |
| SourceRef | 来源引用 |

### 6.2 关键关系

一个事件可以：

- 属于多个事件线；
- 影响多个目标线；
- 发生在一个时间点或时间段；
- 关联一个或多个章节；
- 产生多个信息变化；
- 触发多个风险项。

一条事件线必须：

- 只由正文可见事件块构成；
- 可以有多个离散块；
- 不用虚线或淡色范围表示人物后台状态；
- 通过展开人物条查看人物参与，而不是把人物状态混入事件块边界。

一条人物条必须：

- 归属于某条事件线；
- 绑定一个人物；
- 区分实段和虚段；
- 用实段表示正文已经写出的行为；
- 用虚段表示逻辑存在但正文未写出的后台状态；
- 为虚段提供触发事件或证据来源。

一条目标线必须：

- 有总体目标；
- 有当前注意力点；
- 有最近触发事件；
- 有行动输出；
- 有副作用或代价。

一条信息流必须：

- 有来源；
- 有传播路径；
- 有知情人物；
- 有读者可见位置；
- 有可信性说明。

## 8. 实施阶段建议

### 阶段一：纯文档样板

目标：验证结构是否有用。

内容：

- 建立事件卡模板。
- 建立目标卡模板。
- 建立章节准备包模板。
- 用 E029-E035 做一组手工样板。

成功标准：

- 能看清第十五章开篇该承接什么。
- 能看清会后缓冲期哪些线该进，哪些线该等。
- 能看清张航远、叶依人、莫妮卡、电视台的目标差异。

### 阶段二：半自动检查

目标：用脚本辅助发现明显断裂。

内容：

- 读取事件卡中的事件 ID。
- 检查事件 ID 是否重复或缺失。
- 检查事件是否缺少时间、事件线、目标线或状态。
- 生成 Markdown 风险报告。

成功标准：

- 工具能列出缺字段和重复编号。
- 工具不会修改正式文档。
- 报告可读，能直接服务写作讨论。

### 阶段三：章节准备包生成

目标：写新章前自动整理最小上下文。

内容：

- 输入章节号或事件范围。
- 输出相关事件、目标、信息流和风险。
- 给出本章写作边界。

成功标准：

- 准备包比人工翻多个文档更快。
- 准备包不会把远场内容过度拉入当前章。
- 能明确标出未确认和未同步内容。

### 阶段四：可视化或交互界面

目标：在前面三个阶段证明有效后，再考虑界面。

可选形式：

- 本地 HTML 页面。
- Obsidian 风格 Markdown 图谱。
- 简单命令行菜单。
- CSV/表格导出。

暂不建议一开始做复杂界面。界面早做会把注意力从叙事规则转移到操作体验，容易做成漂亮但无用的资料库。

## 9. 第一批样板范围建议

建议不要从全书开始。第一批样板只覆盖：

- E029 会议结论成为高层定调
- E030 电视台推出民生专区
- E031 三天缓冲期开启
- E032 叶依人认识一线群众的复杂性
- E033 林盼儿双线诱导
- E034 莫妮卡获得家主系信用回升
- E035 张航远发现自己被边缘化

选择这一段的原因：

1. 它正好承接第十四章结果。
2. 它决定第十五章开篇方向。
3. 它包含时间线、事件线、目标线、媒体口径、心理变化和后续冲突。
4. 它足够小，适合验证工具是否有用。

## 10. 第十五章准备包示例目标

如果用本工具准备第十五章，输出应该至少回答：

1. 叶依人此时不是胜利者，而是被名誉和执行机制绑定的人。
2. 莫妮卡不是单纯保护者，而是在把结果转成可控秩序。
3. 电视台不是变善，而是在把爆炸性新闻改造成可管理叙事。
4. 张航远的核心变化不是继续采访，而是失去中心感。
5. 林盼儿适合在会后缓冲期继续诱导张航远，不应突然变成明牌反派。
6. 化学证据线此时可以准备，但不应提前万能解决职业病问题。

## 11. 不做事项

当前阶段不做：

- 不自动重写正文。
- 不自动修改正式大纲。
- 不把规划内容直接标成已确认。
- 不读全书全文做大规模分析。
- 不做复杂数据库。
- 不做网页界面。
- 不做 AI 自动续写。

这些限制是为了先确认工具是否真正帮助叙事判断，而不是制造新的维护负担。

## 12. 成功标准

一个可用的第一版工具，不需要功能很多，但必须做到：

1. 能让人一眼看到某章前后的关键事件和目标变化。
2. 能指出某个新线索为什么应该加入、并入、延后或废弃。
3. 能发现时间、因果、信息流和人物目标上的明显问题。
4. 能明确哪些内容已成文，哪些只是规划，哪些还待确认。
5. 能生成不会污染正式文档的检查报告。
